# AMeDAS気温空間分布解析システム

## 1.特徴
##### ①家庭用PCでもとにかく早い！！
　　250mメッシュ日本全国（7329 x 12057ピクセル）で1か月分の計算（24h x 日数）が30秒弱<br>
　　　　※画像出力を行うとその分時間がかかります<br>
##### ②設定ファイルで動作を変えることができる
　　※複数の設定ファイルを指定して別条件で連続的に実行することもできます<br>
##### ③DEMデータの設定による差し替えで、自動的に解析エリア範囲、メッシュ精度を切り替えるkとができる
　※広域エリア、精細メッシュの場合は当然メモリを必要とします<br>
　　　※日本全国250mメッシュ、北海道50mメッシュであれば16GBのメモリ量で動作可能です<br>
　　　　　※起動時のメモリ消費によります<br>
##### ④設定ファイルによる設定により補間方法、各種解析を切り替えることが可能
以下の補間方法に対応<br>
　　IDW（Inverse Distance Weighted、逆距離加重）法<br>
　　ガウシアンRBF(Radial Basis Function Interpolation) 法<br>
　　Shepard修正IDW（べき乗可変）<br>
　　Barnes法（1Pass）<br>
　　Barnes法（2Pass）<br>
　　TPS（Thin Plate Spline）法<br>
　　※補間方法はAMeDAS観測点密度が低く理論通りの精度が出ずにBarnes（1Pass）が一番バランス的に良かった<br>
一定閾値以下／以上の確率分析<br>
　　16閾値まで設定可能<br>
エリア解析<br>
　　エリア指定による統計解析<br>
　　　※最大32、32以上の場合は複数の設定ファイルで実行<br>
重心分析<br>
　　気温境界面の重心分析<br>
　　全エリアだけでなく、指定点を含む境界面の重心分析も可能<br>
時間軸でのベクトル解析（オプティカルフロー）<br>
##### ⑤各種統計解析の結果テキストファイル出力
　　全域／エリア別で分析<br>
　　気温統計情報、気温ヒストグラム、閾値以下／以上の確率ヒストグラムなど<br>
##### ⑥解析結果のGeoTIFF、PNG出力
　　補間気温分布、等温線、閾値以上／以下確率分布など<br>
　　※計算処理よりも時間がかかるので以下の手順が便利<br>
　　　　1回目：画像出力なしで統計処理、気温ヒストグラム分析を行い、全エリアの気温範囲を確認<br>
　　　　2回目：短期間でPNG出力の気温レンジを調整<br>
　　　　3回目：PNG画像出力走行<br>
　　　　　　※気温気温、等温線表示がおすすめ<br>
　　　　　　※出力した画像をmp4化<br>
　　　　　　　　※これを眺めていると特徴的なエリアが見えてくる<br>
　　　　4回目：エリア解析を行う場所を決定<br>
　　　　　　※32エリアまで設定可能、それ以上の場合は複数の設定ファイルを作り実行<br>
　　　　5回目：詳細に見たい期間に絞ってGeoTIFF出力<br>
##### ⑦エリア解析におけるポリゴン境界対応
　　緯度経度指定の方形解析だけでなく、GISデータ（SHPファイル）を使用したポリゴンエリア解析も対応<br>

## 解析に使用するAmeDASデータ
2000年以前のバイナリフォーマット<br>
<格納先パス>\1979<br>
<格納先パス>\1980<br>

## DEMデータ変換
### 国土地理院10mメッシュ（10B）の場合
以下の手順は（株）エコリスさんのツールやQGISでQuickDEM4JPプラグインで行っても良い<br>
　※ダウンロード漏れの確認のため1次メッシュでファイルを作りたかったのでPythonコードで変換<br>
①必用なエリアのメッシュデータをxmlが格納されているzipファイルを一か所のパスに格納<br>
②conv_10m_dem.pyのsrc_dir、cnv_dirを修正<br>
③conv_10m_dem.pyを実行<br>
　　Python conv_10m_dem.py<br>
　　⇒cnv_dirに1次メッシュ毎のGeoTIFFが作成される<br>
③VRT作成<br>
以下のコマンドをパスを変更して実行<br>
```
# 5A（5m）だけのVRT
# 10Bデータで除去した2次メッシュを含む1次メッシュ
gdalbuildvrt -allow_projection_difference -srcnodata -9999 U:\DEM5A\5A_all.vrt U:\DEM5A\TIFF\*.tif
# 10B（10m）だけのVRT
# 2次メッシュ単位で異常データを除去
gdalbuildvrt -allow_projection_difference -srcnodata -9999 U:\DEM10B\10B_all.vrt U:\DEM10B\TIFF1\*.tif
# 2つのVRTをマージする
gdalbuildvrt -allow_projection_difference -srcnodata -9999 -vrtnodata -9999 ^
  U:\DEM\all_japan.vrt ^
  U:\DEM5A\5A_all.vrt ^
  U:\DEM10B\10B_all.vrt
```
　　基準地図情報の5mメッシュ（5A）が利用できるところは5mメッシュ（5A）を利用し、データがない部分は10mメッシュ（10B）を利用する<br>
　　測位法が違うので無理やり一体化（10mメッシュでは差がないため）<br>

### 切り出し方
基本の形<br>
```
gdal_translate -of GTiff -r average ^
  -projwin [左上経度] [左上緯度] [右下経度] [右下緯度] ^
  -tr 0.000333333333 0.000333333333 ^
  -co COMPRESS=LZW -co TILED=YES ^
  U:\DEM\all_japan.vrt U:\DEM\output.tif
```

## 解析結果テキストファイル出力
### threshold_stat_hourly.txt
その時刻に、「閾値を超えたピクセルが何点あったか・全陸地に対して何%か」を閾値×エリア別に記録<br>
1行 = 1時刻スライス　<br>
| ブロック | 内容 |
| --- | --- |
| 年月日時刻 / 年 / 月 / 日 / 時 | タイムスタンプ（5列） |
| 比率ブロック | 気温が閾値以下となるピクセル数の陸地ピクセル数の比率 % |
| ピクセル数ブロック | 気温が閾値以下となるピクセル数 |

　→ 閾値c × (全域 + エリア1..N) の順に並ぶ<br>


 ### threshold_<lo/hi>_stat_monthly.txt
  1行 = 1か月
| 列 | 内容 |
| --- | --- |
| 年 / 月 / Steps | タイムスタンプ + その月のスライス数 |
| 発生時間数 | 閾値以下となるメッシュが1ピクセル以上存在した時間数（時） |
| 平均メッシュ数 | 閾値以下となるメッシュ数の月平均（area_sum ÷ Steps） |
| 最大メッシュ数 | 閾値以下となるメッシュ数の月最大（そのスライスでの最大） |
| 消滅回数 | 閾値以下となるメッシュ数が「あり→なし」になった回数（消滅） |
| 出現回数 | 閾値以下となるメッシュ数が「なし→あり」になった回数（出現） |
　
  → 上記5項目が 閾値c × (全域 + エリア1..N) の順で並ぶ<br>
  閾値以下はthreshold_lo_stat_monthly.txt、閾値以上はthreshold_hi_stat_monthly.txtに出力される。<br>
消滅回数/出現回数の意味補足: 閾値以下（あるいは以上）の断続性を表す。<br>
値が大きい = 短命なイベントが頻発。<br>
小さい = 一度出現したら長続きする収束域。<br>
結果の見方の補足：<br>
| 観点 | 全域 | 個別エリア |
| --- | --- | --- |
| 発生時間数=744 | 月中全時刻で存在 | 発生時間数<744ならその時間は当エリア内に対象なし |
| 消滅回数=0 | 全月で途切れなし | 途切れた回数 |
| 最大メッシュ数≒平均メッシュ数 | 変動小さい（常に広域分布） | ピーク時と平均の差が大きい = 急激な拡大縮小あり |

## 留意点
### 重心点移動に関して
\*_bound_cen.txt、\*_poi_centroid.txtの出力結果に関して<br>
[RUN]PARALLEL_SLICESが0の場合と1以上の場合に1ピクセル程度の差異がでます。<br>
これは計算方法の違い（複数時間を並列で処理する際の）により発生するものになります。<br>
### 整数モードに関して
設定ファイルの以下の設定で近傍点嘆息条件の一部をfloatから整数化することが可能です。<br>
[RUN]セクションUSE_NBR_FLOAT<br>
0:すべて整数　　⇒　整数化可能なものをすべて整数化<br>
1:SRCのみfloat　⇒　加重情報を整数化<br>
2:Wのみfloat　　⇒　近傍点距離情報、高度情報を整数化<br>
3：全てfloat　　⇒　標準<br>

精度的には3（全てfloat）が良いが、3⇒2⇒0の順でメモリ消費を抑えることができます。<br>
近傍点情報が解析条件によらず最大メモリ消費となる部分のため、解析環境の搭載メモリ量によっては広範囲解析あるいは詳細メッシュ解析では必要になる場合があると思います<br>
参考1:USE_NBR_FLOAT=3(全てfloat) ⇒ 2(一部float)の変更により<br>
気温の統計解析（\*temp_stat_hourly.txt、\*temp_hist_hourly.txt）で評価した一例として、以下の結果となっています。<br>
　北海道250mメッシュ、1999～2001年、1月条件<br>
1.temp_stat_hourly.txtで平均、最低、最高で±0.03℃を超える差はない　⇒　0.1℃の誤差はない<br>
2.temp_stat_hourly.txtで中央値±0.1℃差が186件/37944<br>
　　※中央値の導出が0.1℃BINのヒストグラム解析から簡易的に導出しているため<br>
3.temp_hist_hourly.txtでヒストグラム%で±0.1%以上の差が4件/2124864件（全て±0.2%未満）<br>
4.4000x4000ピクセル範囲、全て陸地領域の解析では約1GBのメモリ節約（全てfloatの場合は約6.9GB）<br>
　　※50mメッシュで200km四方<br>
　　　　⇒ 16GBメモリ搭載PCの限界レベル<br>

参考2:USE_NBR_FLOAT=3(全てfloat) ⇒ 0(NBR全て整数化)の変更により<br>
1.temp_stat_hourly.txtで平均、最低、最高で±0.05℃差は5件/37944（全域のみ）<br>
2.temp_stat_hourly.txtで中央値±0.1℃差が105件/37944件<br>
3.temp_hist_hourly.txtでヒストグラム%で±0.1%以上の差が21件/2124864件（全て±0.2%未満）<br>
4.4000x4000全て陸地領域の解析では約2.1GBのメモリ節約（全てfloatの場合は約6.9GB）で非常に効果が大きい<br>
